# 画廊滚动与分类设计

## 目标

修复瀑布画廊在窄窗口下失去内部滚动的问题，并把无序长列表改成更容易浏览的分组画廊。

## 现状问题

- 画廊在响应式布局下会失去固定可视高度，滚动区域被内容整体撑开。
- 所有图片直接堆在一个瀑布流里，数量一多就难以定位最近输出。
- 用户已经要求不再生成侧边 `json` 结果文件，因此分类方案应优先依赖图片现有信息，而不是新的 sidecar 文件。

## 方案

### 1. 滚动修复

- 为画廊面板单独同步可用视口高度。
- 保持 `gallery-scroll-region` 作为唯一滚动容器。
- 让自定义滚动条继续跟随这个滚动容器，而不是跟随整页高度。

### 2. 分类浏览

- 画廊改为“按日期分组”的结构。
- 组头显示相对日期文案和图片数量，例如“今天 / 昨天 / 2026-04-22”。
- 顶部增加轻量筛选条，先提供稳定可用的时间范围筛选：`全部`、`今天`、`近 7 天`、`更早`。
- 每个分组内部仍保持瀑布流，避免一次性把 600+ 张图揉成一整屏。

### 3. 数据来源

- 分组和筛选只依赖 `createdAt`。
- 继续允许已有内存态画廊项携带更多展示信息，但新浏览结构不依赖额外元数据文件。

## 组件边界

- `lib/gallery-organizer.mjs`：画廊排序、时间筛选、日期分组。
- `public/index.html`：画廊筛选条和分组容器。
- `public/app.js`：画廊筛选状态、分组渲染、滚动高度同步。
- `public/styles.css`：筛选条、分组头、响应式滚动容器样式。

## 验证

- 单元测试验证分组/筛选逻辑。
- 浏览器验证：
  - 画廊内部可以滚动；
  - 筛选条可切换；
  - 日期分组正常显示；
  - 大量图片下不再整页摊开。
